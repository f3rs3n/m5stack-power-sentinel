NUT Ledcards: stato attuale

Mappa compatta di condition, visualClass, priority/hero e soglie delle 5 card NUT del firmware CoreS3. La visual class è solo colore renderizzato; la condition canonica resta separata.

5 cards: Battery, Runtime, Load, Input, NUT telemetryState: live / partial / stale / unavailable condition: healthy / warning / critical / stale / unavailable missing live metric: value --, class unavailable/purple per card

1. Condition modulo NUT lato backend

PrioritàConditionTrigger
1criticalwould_shutdown oppure runtime_seconds < 60 oppure load_percent >= 90
2staleups.stale true e UPS ancora disponibile
2unavailableups.stale true e UPS non disponibile; oppure condition assente/non canonica nel page model

4healthyNessun trigger sopra

2. Telemetry state e missing data

StatoRegolaEffetto card
unavailable!upsAvailableTutte le card: value --, stateText UNAVAILABLE, stateClass unavailable, visualClass purple
staleoffline || upsStaleTutte le card: value --, stateText transportStatus se presente altrimenti STALE, visualClass gray
partialTelemetry live ma una metrica manca: battery/runtime/load/input/clientSolo la metrica mancante diventa UNAVAILABLE/purple; il modello segnala hasMissingMetrics
liveTutte le metriche chiave presentiOgni card applica le soglie sotto

Stale/unavailable battono le soglie delle singole metriche: prima si decide la freschezza, poi le card.

3. Hero / priority policy

  1. NUTtelemetry missingSe la sorgente è stale o unavailable, promuove la card NUT.
  2. RuntimeonBattery && runtime < 60sRuntime critico su batteria: massima urgenza operativa.
  3. BatteryonBattery && (lowBattery || battery < 20%)Batteria bassa su batteria; se non è runtime <60s, vince Battery.
  4. RuntimeonBattery && runtime knownSu batteria, Runtime è il default anche se non critico.
  5. Loadload >= 70%Alto carico quando non c'è già un evento batteria più rilevante.
  6. Input0V < input < 210VInput marginale/basso quando online.
  7. BatterydefaultLinea sana: Battery è la hero di default.

Touch override: una card toccata resta hero per 60s. Gli swap automatici rispettano cooldown 5s.

4. Regole da tenere a mente

ConcettoStato attuale
Visual conditionNel codice è visualClass: green/yellow/orange/red/blue/purple/gray. Non è una condition API.
Visual depthBlue e yellow possono essere profondità estetica healthy/informational; orange e red restano attenzione/urgenza.
Shutdown relevantTrue solo se telemetry presente, onBattery && lowBattery. Backend would_shutdown è OB+LB.
Module vs cardIl backend calcola la condition del modulo; il page model la conserva se canonica e calcola card stateClass/visualClass separatamente.
Card orderOrdine stabile: Battery, Runtime, Load, Input, NUT. La hero può promuovere una metrica senza cambiare la semantica delle card.
